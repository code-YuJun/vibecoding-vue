# 文档规范

## 基本原则（NON-NEGOTIABLE）

文档应做到：

- 内容准确。
- 结构清晰。
- 简洁易懂。
- 与代码保持同步。

禁止：

- 编写无意义文档。
- 文档与代码不一致。
- 长时间保留失效文档。

---

# 中文规范（NON-NEGOTIABLE）

所有项目文档统一使用中文。

包括：

- README
- OpenSpec
- Rules
- Skills
- Markdown
- 代码注释

代码中的：

- 变量
- 函数
- 类
- Interface
- Type
- Enum

统一使用英文。

# JSDoc 规范

公开函数、公共模块、工具函数建议使用 JSDoc。

例如：

```ts
/**
 * 获取用户列表
 *
 * @param params 查询参数
 * @returns 用户列表
 */
export function getUserList(params: GetUserListParams) {}
```

应说明：

- 功能
- 参数
- 返回值

不要描述代码本身。

例如：

❌

```ts
/**
 * 定义一个变量
 */
const count = 0;
```

---

# 文件头注释

对于以下文件建议增加文件说明：

- composables
- utils
- http
- stores
- plugins

例如：

```ts
/**
 * 用户相关接口
 *
 * 统一维护用户模块所有 HTTP 请求。
 */
```

普通 Vue 页面、组件无需强制添加文件头注释。

---

# 复杂逻辑注释

以下情况建议增加注释：

- 复杂算法
- 特殊业务规则
- 多条件判断
- 第三方兼容处理
- 性能优化

例如：

```ts
// 登录成功后需要同步刷新权限缓存，避免页面权限异常
```

不要写：

```ts
// 判断 if
if (...)
```

---

# TODO 规范

未完成能力必须使用：

```ts
// TODO: 等待接口完成

// TODO: 等待设计稿

// TODO: 待产品确认
```

禁止：

```ts
// fix

// later

// test
```

# README 规范

项目 README 建议包含：

- 项目介绍
- 技术栈
- 安装方式
- 启动方式
- 打包方式
- 目录结构
- 环境变量
- 开发规范

保持 README 与项目一致。

---

# OpenSpec 文档

需求开发统一使用 OpenSpec。

建议：

```
proposal.md
tasks.md
spec.md
```

要求：

- 描述业务需求。
- 描述验收标准。
- 不记录实现细节。
- 保持与代码同步。

---

# Rules 文档

Rules 用于约束 AI 的开发行为。

要求：

- 描述项目规范。
- 不记录实现流程。
- 不重复 Skill 内容。
- 一个 Rule 负责一个领域。

例如：

```
05-API规范

06-路由规范

07-Pinia规范
```

---

# Skills 文档

Skills 用于指导 AI 如何完成任务。

建议统一结构：

```
使用场景

前置检查

Step 1

Step 2

Step 3

输出结果

Checklist
```

要求：

- 描述执行流程。
- 引用对应 Rule。
- 不重复 Rule 内容。

---

# Markdown 规范

统一：

- 一级标题仅一个。
- 标题层级不超过四级。
- 使用 Markdown Table。
- 使用代码块标注语言。

例如：

````md
```ts
const count = ref(0);
```
````
