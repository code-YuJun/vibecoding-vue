# 测试规范

## 基本原则（NON-NEGOTIABLE）

所有新增或修改的代码必须保证：

- TypeScript 类型检查通过。
- ESLint 检查通过。
- 项目能够正常构建。
- 不影响已有功能。

禁止：

- 提交存在 TypeScript 类型错误的代码。
- 提交存在 ESLint 错误的代码。
- 为通过检查而随意关闭规则。

---

# TypeScript

新增代码必须：

- 无类型报错。
- 不使用 `any`（特殊情况除外）。
- 类型定义完整。

推荐：

```ts
interface User {}

type UserStatus = 'enable' | 'disable';
```

---

# ESLint

所有代码必须通过：

```
npm run lint
```

禁止：

```ts
// eslint-disable-next-line
```

除非确有必要，并说明原因。

---

# 构建检查

提交前至少保证：

```
npm run build
```

能够正常通过。

不得提交无法构建的代码。

---

# 单元测试

项目已引入测试框架时：

建议为以下内容编写测试：

- 工具函数（utils）
- Composables
- Pinia Store
- 复杂业务逻辑

普通页面组件无需强制编写测试。

---

# API 测试

接口开发完成后建议验证：

- 请求参数正确。
- 返回类型正确。
- 异常处理正常。

接口定义应以 Apifox 为准。

不得自行猜测接口字段。

---

# 页面验证

新增页面建议检查：

- 页面正常打开。
- 路由正常跳转。
- 表单正常提交。
- Element Plus 组件显示正常。
- 响应式布局正常。

---

# AI 开发约定

AI 完成功能后，应主动检查：

- 是否存在 TypeScript 报错。
- 是否存在 ESLint 报错。
- 是否影响已有代码。
- 是否存在未使用变量。
- 是否存在未引用文件。

如发现问题，应优先修复，而不是忽略。

---

# Checklist

- [ ] TypeScript 类型检查通过。
- [ ] ESLint 检查通过。
- [ ] 项目构建成功。
- [ ] 类型定义完整。
- [ ] 未引入无用代码。
- [ ] 关键业务逻辑已验证。
- [ ] 接口类型与 Apifox 保持一致。
